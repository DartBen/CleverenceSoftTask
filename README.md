# CleverenceSoftTask

Тестовое задание. 
Решение трёх задач: компрессия строк, асинхронный счётчик с Reader-Writer Lock и консольная утилита для стандартизации лог-файлов.

## Задача 1: Компрессия строк

`StringCompressionUtility` — статический класс для сжатия и восстановления строк по алгоритму Run-Length Encoding.

### Возможности
* **Компрессия:** группы одинаковых символов заменяются на формат `<символ><количество>`. Если символ один — количество не указывается.
* **Декомпрессия:** восстановление исходной строки из сжатой формы.
* **Производительность:** используются `StringBuilder` и `ReadOnlySpan<char>` для минимизации аллокаций в куче. Проверка цифр выполняется через `HashSet<char>` за O(1).
* **Тестирование:** покрыт модульными тестами для типовых и граничных сценариев.

### Пример
```
Исходная строка:  aaabbcccdde
Сжатая строка:    a3b2c3d2e
```

## Задача 2: Асинхронный счётчик с Reader-Writer Lock

`AsyncCounterService` — статический класс со счётчиком `count` (int), к которому параллельно обращаются множество клиентов.

### Гарантии
1. **Чтение** (`GetCount`) — разрешено параллельно для любого числа клиентов.
2. **Запись** (`AddToCount`) — строго последовательна: только один писатель в момент времени.
3. **Взаимное исключение**: пока идёт запись, читатели ждут; пока есть читатели, писатели ждут.

### Реализация
Реализован собственный `AsyncReaderWriterLock` на базе `SemaphoreSlim` с приоритетом писателей (writer-preference):
* Два семафора: `_writerSemaphore` (эксклюзивный доступ писателя) и `_readerGateSemaphore` (ворота для читателей).
* Атомарные счётчики `_readerCount` и `_writerCount` через `Interlocked`.
* Если писатель встал в очередь, новые читатели блокируются на `_readerGateSemaphore` — это защищает писателей от голодания при постоянном потоке читателей.
* Методы возвращают `IDisposable` через `struct` для минимизации аллокаций в куче.

### Пример использования
```csharp
await AsyncCounterService.AddToCount(5);
var count = await AsyncCounterService.GetCount();
```

## Задача 3: Стандартизация лог-файлов

Консольная программа для обработки лог-файлов, содержащих записи в двух разных форматах.

### Функциональность
* Читает входной лог-файл построчно.
* Определяет формат каждой строки (Format 1 или Format 2) с помощью регулярных выражений.
* Преобразует строки в единый стандартный формат с разделителем-табуляцией.
* Нормализует уровни логирования: `INFORMATION` → `INFO`, `WARNING` → `WARN`.
* Если поле "ВызвавшийМетод" отсутствует — подставляется `DEFAULT`.
* Невалидные строки записываются в отдельный файл `problems.txt`.

### Аргументы командной строки
| Аргумент | Алиас | Описание | По умолчанию |
|---|---|---|---|
| `--input` | `-i` | Путь к входному лог-файлу | `input.log` |
| `--output` | `-o` | Путь к выходному файлу | `output.log` |
| `--problems` | `-p` | Путь к файлу с невалидными строками | `problems.txt` |

### Пример запуска
```bash
ThirdTask.exe -i input.log -o output.log -p problems.txt
```
или 

```PowerShell
.\ThirdTask.exe -i input.log -o output.log -p problems.txt
```

### Справка
```bash
ThirdTask.exe --help
```
или 
```PowerShell
ThirdTask.exe --help
```

## Ключевые технические решения

1. **DI-контейнер через `Microsoft.Extensions.Hosting`** — сервисы регистрируются через стандартный DI, что упрощает тестирование и замену реализаций.
2. **Фабрика парсеров** (`ILogParserFactory`) — создание парсеров инкапсулировано, что позволяет добавлять новые форматы без изменения `LogProcessor`.
3. **Result-паттерн** (`LogMetadata`) — ошибки парсинга обрабатываются без исключений, что повышает производительность при большом количестве невалидных строк.
4. **Regex с именованными группами** — парсер валидирует наличие обязательных групп (`date`, `time`, `loglevel`, `message`, `invoker`) при создании, что гарантирует корректность паттернов.
5. **AsyncReaderWriterLock на семафорах** — компактная реализация с приоритетом писателей без использования `lock`, что снижает contention при высокой нагрузке.
